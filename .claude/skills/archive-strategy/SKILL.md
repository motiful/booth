# Progress 文档归档策略

## 适用范围

`.claude/progress.md`（进度追踪文档）

## 归档原则

### 1. 何时归档

当一个阶段（Phase）或一组功能完成时：
- 已完成的详细内容 → 移动到 `.claude/archive/` 目录
- 当前 progress 文件 → 保留精简摘要

### 2. 归档内容（原文保留）

归档文件必须**原文完整保留**，不做任何删减：
- 完整的任务清单和状态
- 所有的验收记录
- 详细的更新日志
- 发现的问题和解决方案

**命名格式**: `progress-{阶段名}-{日期}.md`

**示例**: `progress-phase1-to-2.8-2026-03-11.md`

### 3. 主文件精简（保留上下文）

精简后的主 progress 文件需要保留：

#### 必须保留
- 未完成任务的完整信息（不删减）
- 未完成任务依赖的上下文信息
- Architecture 段落（活文档，非历史）
- Key Design Decisions 表
- Phase Status 表
- File Map

#### 可以精简
- 已完成阶段：保留**摘要**（做了什么、关键决策、归档位置）
- 已完成任务：保留**一句话总结**
- 详细的更新日志：只保留最近的，历史的随归档移走

#### 删除
- 一次性验证 checklist（E2E Verification Checklist 等）——已验证的功能不需要留着

### 4. 精简模板

```markdown
## [已完成阶段名] — 已归档

**归档文件**: `.claude/archive/xxx.md`
**摘要**: 做了什么（3-5 行）
**关键决策**: 影响后续任务的（1-3 条）
```

## 目录结构

```
.claude/
├── progress.md             # 当前进行中的进度
├── archive/                # 已完成阶段的归档
│   └── progress-phase1-to-2.8-2026-03-11.md
└── skills/
    └── archive-strategy/
        └── SKILL.md        # 本文件
```

## 注意事项

1. **不要删除历史** — 归档是移动，不是删除
2. **保留依赖链** — 未完成任务需要的上下文必须保留在主文件
3. **归档后检查** — 确保主文件仍然可以独立阅读
4. **零丢失** — 所有 `[ ]` 未完成项必须在主文件中
